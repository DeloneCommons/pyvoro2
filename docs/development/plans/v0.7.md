## Intended outcome

v0.7.0 should make pyvoro2 a dependable foundation for chemvoro and for later
inverse weighted-tessellation methods.

At release, a downstream package should be able to:

1. define 2D or 3D sites, domains, and external IDs without relying on private
   parsing or record-order assumptions;
2. compute a power diagram directly from mathematical power weights;
3. inspect cells, cell measures, realized boundaries, periodic image labels,
   empty-cell state, and diagnostics through an aligned result contract;
4. fit weights from separator observations through a preferred, math-aligned
   inverse API;
5. distinguish global gauge, unidentified offsets between disconnected
   observation components, algebraic residuals, and realized geometry;
6. export documented records or reports without depending on solver internals;
7. continue running documented v0.6.3 `pyvoro2.powerfit` workflows through a
   tested compatibility path.

The release should stabilize responsibilities and access contracts, not freeze a
generic inverse framework before a second observation family exists.

## Baseline and motivation

The v0.6.3 baseline already provides substantial research functionality:

- explicit 3D and planar 2D forward namespaces;
- bounded and periodic domains;
- standard and power/Laguerre tessellations;
- periodic face/edge image labels;
- diagnostics, normalization, geometric annotations, `locate`, and
  `ghost_cells`;
- separator-observation fitting with graph/connectivity diagnostics, robust and
  hard constraints, problem export, result packaging, realization matching,
  active-set refinement, and report generation.

The main limitations are architectural rather than a lack of algorithms:

- 3D normally returns raw cell records, while planar computation has an
  optional `PlanarComputeResult`;
- forward power computation is radius-first even though inverse mathematics is
  weight-first;
- the public inverse vocabulary is centered on `powerfit` and pairwise
  constraints rather than one observation family within a broader inverse
  layer;
- top-level exports expose many separator-specific implementation names;
- algebraic, realization, and solver diagnostics are rich but not yet organized
  around a long-lived cross-observation vocabulary;
- chemvoro would still need to understand some representation and result-shape
  details that should belong to pyvoro2.

Adding prescribed cell measures before resolving these boundaries would either
duplicate geometry and result handling or prematurely freeze an over-general
API. v0.7 therefore focuses on forward/result and separator stabilization.

## Accepted constraints

This plan does not reopen the following accepted decisions:

- [ADR 0001 — Project scope and layering](../decisions/0001-project-scope-and-layering.md):
  pyvoro2 is a forward and inverse weighted-tessellation package; the forward
  core remains independently useful.
- [ADR 0002 — Weights, radii, gauge, and component offsets](../decisions/0002-weights-radii-and-gauge.md):
  power weights are mathematical, backend radii are a shifted representation,
  and disconnected observation-component offsets are not additional harmless
  global gauges.
- [ADR 0003 — Compatibility-first evolution](../decisions/0003-compatibility-first-evolution.md):
  documented v0.6.3 workflows receive a migration path rather than an abrupt
  rewrite.
- The 2D and 3D namespaces remain explicit where capabilities differ.
- The fixed-observation inverse solve remains separate from realization-aware
  outer algorithms.
- Chemistry-specific data and policy remain in chemvoro or other downstream
  packages.

## Scope

Required v0.7 outcomes are:

- a reviewed public API and result-schema inventory;
- a documented common forward-result access contract across 2D and 3D;
- a compatible structured 3D result route and aligned planar result behavior;
- direct `weights=` input for forward power computation in both dimensions;
- explicit representation-shift metadata where weights are converted to radii;
- a preferred inverse organization and separator-observation terminology,
  subject to decision D-07-1;
- tested compatibility for existing `pyvoro2.powerfit` and documented top-level
  workflows;
- clearer separation of fitted state, observation residuals, algebraic graph
  diagnostics, realized geometry, and solver status;
- inspectable incidence/Laplacian problem ingredients without requiring SciPy
  as the public data model;
- a chemvoro-shaped downstream contract test or example;
- migration documentation, lifecycle labels, release notes, and complete
  release validation.

A sparse quadratic execution path is desirable but conditional: it should be
included only if a benchmark and dependency review show a clear benefit without
complicating the core API.

## Non-goals

v0.7 does not include:

- prescribed area or volume inversion;
- a public generic `ObservationBlock` protocol;
- mixed separator and cell-measure objectives;
- site-coordinate optimization;
- centroid, section, anisotropic, spherical, or non-Euclidean observations;
- arbitrary user-defined nonlinear objective callbacks;
- GPU acceleration;
- a persistent mutable tessellation container;
- a general computational-geometry abstraction over Voro++;
- guaranteed convergence of the realization-aware active-set loop;
- removal of the archived v0.6.3 API or recomputation of manuscript results.

Planar oblique-periodic support is also not a v0.7 requirement unless a concrete
downstream need independently justifies it.

as the first design issues after plan activation. The third must be complete
before release. The fourth is a documentation-quality decision and is not a

| ID | Decision | Release impact | Expected resolution | Status |

### D-07-1 — Preferred inverse namespace

Realistic options:

1. **Adopt `pyvoro2.inverse` as the preferred namespace.**
   `pyvoro2.powerfit` remains a compatibility and convenience facade.
2. **Keep `pyvoro2.powerfit` as the preferred public namespace.**
   Future observation families would either join it or use separate modules.
3. **Rely mainly on top-level exports.**
   This preserves short imports but leaves ownership and future organization
   unclear.

Preliminary recommendation: adopt option 1. It gives separator, cell-measure,
and later mixed work one intelligible home while preserving current imports.
The first implementation can be re-exports and aliases; it does not require an
immediate internal rewrite.

Questions to settle:

- whether `pyvoro2.powerfit` remains fully documented or receives a concise
  compatibility-oriented reference;
- whether top-level inverse re-exports remain indefinitely, become
  compatibility-only, or enter deprecation later;
- whether active-set functionality belongs directly in the preferred namespace
  or in an explicitly experimental submodule;
- which names are aliases and which are new semantic abstractions.

### D-07-2 — Common forward-result design

Realistic options:

1. **One dimension-neutral `TessellationResult`.**
   A single class carries cells, metadata, diagnostics, and optional normalized
   output for 2D and 3D.
2. **Aligned dimension-specific result containers.**
   A spatial result and `PlanarComputeResult` expose the same documented core
   properties while retaining dimension-specific normalized structures.
3. **Adapters over existing return values only.**
   Raw returns remain primary and helper functions provide common access.

Preliminary recommendation: prototype option 2 first. It preserves explicit
2D/3D capabilities, can keep existing raw return behavior, and avoids freezing a
public protocol or inheritance hierarchy before the needs of cell-measure
inversion are known. Shared access should be documented by behavior rather than
by a speculative abstract base class.

The prototype comparison must answer:

- whether raw list/tuple returns remain the default or become an explicit
  compatibility route;
- how domain, dimension, IDs, mode, input weights, representation shift, and
  diagnostics are stored;
- how cell area/volume and boundary length/area are exposed without eager
  computation of unrequested geometry;
- how normalization output fits without unions that are difficult to use;
- how empty sites are represented and associated with original IDs;
- whether result objects should be immutable;
- how record export remains JSON-friendly.

### D-07-3 — Public lifecycle inventory

The release needs an explicit inventory, not a general statement that “the API
is stabilized.” The inventory should cover:

- top-level `pyvoro2` exports;
- `pyvoro2.planar` exports;
- `pyvoro2.powerfit` exports;
- the proposed preferred inverse namespace;
- compute return modes and record keys;
- diagnostic and report fields;
- defaults with scientific meaning.

A preliminary classification to test, not yet a decision:

- **stable candidates:** mature domain classes; site/ID association; standard
  and power geometry semantics; periodic shift conventions; current raw cell
  records that must remain available;
- **provisional candidates:** new structured result routes, direct weight-first
  result metadata, preferred inverse names, and new algebraic operator views;
- **experimental candidates:** self-consistent active-set refinement and any
  newly introduced sparse solver backend whose convergence/performance scope is
  still being evaluated;
- **compatibility-only candidates:** broad top-level inverse re-exports and
  historical names replaced by clearer preferred terminology;
- **internal:** backend and solver implementation modules not explicitly
  documented as public.

A dedicated inventory page should be created during WP-01 and finalized in
WP-07. The final classifications require maintainer approval.

### D-07-4 — Theory figures and literature links

The current theory pages are self-contained, but three original diagrams would
substantially improve accessibility:

1. the two-site separator law and weight difference;
2. a three-site cycle-consistency example;
3. the split between algebraic fitting and realized-boundary checking.

Preliminary recommendation: add small original SVG diagrams under `docs/assets/`
after the API decision work, provided they can be maintained independently of
manuscript layout. Link the manuscript only when a stable public preprint or
publication URL exists. Literature links should support the explanation rather
than turn the package docs into a second review article.

This work may be deferred without blocking v0.7 if the manuscript and core API
work have higher priority.

### WP-03 — Weight-first forward power API

**Purpose:** let users compute the mathematical power diagram without manually
choosing a backend radius representation.

Deliverables:

- add direct `weights=` input to spatial and planar power computation;
- retain `radii=`;
- reject simultaneous `weights=` and `radii=` unless a later explicit policy is
  approved;
- centralize weight-to-radius conversion outside separator-specific solver
  internals;
- record the common representation shift in structured result metadata;
- document the invariance of the diagram under one global weight shift;
- make empty-input and finite-value behavior explicit.

Compatibility requirements:

- existing `radii=` calls and numerical outputs remain unchanged;
- no backend-shifted radius is presented as a unique scientific quantity;
- weight input must not apply componentwise shifts.

Validation:

- weight and radius calls agree after a known common shift;
- adding a constant to all weights leaves cells, measures, adjacency, and
  periodic shifts unchanged within tolerance;
- conflicting input paths fail clearly;
- extreme weights and empty cells receive deterministic handling;
- 2D/3D, bounded/periodic coverage.

Acceptance criteria:

- users can compute a power diagram from weights without importing
  `powerfit`;
- result metadata reports the representation shift when applicable;
- compatibility and gauge-invariance tests pass.

Acceptance criteria:

### WP-05 — Separator result layering and algebraic operator access

**Purpose:** make the mathematical structure inspectable and prepare result
vocabulary for later observation families.

Deliverables:

- map existing `PowerWeightFitResult`, problem export, realization diagnostics,
  and active-set results onto distinct conceptual layers:
  - fitted state and identification metadata;
  - objective contributions;
  - observation targets, predictions, and residuals;
  - algebraic graph/operator diagnostics;
  - realized geometry and requested-image matching;
  - solver and active-set path diagnostics;
- expose incidence rows, implied edge-difference targets, edge weights,
  connected components, normal-system right-hand side, and Laplacian/operator
  metadata through a public inspectable object or view;
- provide dense NumPy conversion and an optional SciPy conversion without making
  SciPy types the only public representation;
- use explicit global-gauge and component-offset terminology;
- preserve JSON-friendly records and reports.

Compatibility requirements:

- existing result fields and report builders remain usable;
- a new layered view should prefer composition/adapters over duplicating large
  arrays or changing numerical behavior;
- realization remains optional and is not implied by a successful algebraic
  fit;
- active-set results remain clearly heuristic/experimental where appropriate.

Validation:

- algebraic operator data reproduce current predictions and quadratic normal
  equations;
- disconnected and confidence-zero rows produce correct component diagnostics;
- periodic parallel observations remain distinct multigraph rows;
- existing report serialization remains consistent;
- infeasible hard restrictions retain contradiction witnesses.

Acceptance criteria:

- users can inspect the effective graph and quadratic operator without reading
  private solver code;
- result documentation states which scientific question each layer answers;
- later cell-measure results can reuse the vocabulary without depending on
  separator-specific class names.

### WP-06 — Sparse quadratic benchmark and optional execution path

**Purpose:** determine whether sparse graph linear algebra materially improves
real separator workloads.

This package is conditional and is not a v0.7 release blocker unless later
approved as required.

Deliverables:

- benchmark current dense assembly/solve on small, medium, and larger
  observation graphs;
- prototype sparse incidence/Laplacian assembly using an optional SciPy extra;
- define a crossover or explicit solver-selection policy;
- compare numerical results and component/gauge handling with the dense path;
- assess factorization reuse in active-set iterations when the graph is
  unchanged.

Acceptance criteria for inclusion:

- a documented workload shows a meaningful time or memory benefit;
- dense and sparse results agree within justified tolerances;
- absence of SciPy preserves a complete supported path;
- solver/backend metadata identifies the selected path;
- dependency and packaging costs are acceptable.

If these criteria are not met, archive the benchmark result and defer the sparse
path without blocking v0.7.

## Compatibility and migration contract

The default v0.7 policy is:

- keep existing `radii=` forward calls;
- keep existing raw cell return routes;
- keep documented `pyvoro2.powerfit` imports;
- keep documented top-level inverse imports through v0.7;
- keep current record keys and scientific meanings unless a specific migration
  is approved;
- introduce preferred paths additively where possible;
- use adapters or aliases rather than duplicate numerical implementations;
- avoid deprecation warnings without a documented replacement and removal
  horizon;
- preserve the exact v0.6.3 paper archive as the authority for published
  numerical results.

Pre-1.0 status allows deliberate breaking changes, but this plan does not treat
that permission as a reason to break working scientific code.

## Validation strategy

### Forward matrix

Required representative coverage:

| Dimension | Domain | Mode | Result route | Additional checks |
|---|---|---|---|---|
| 2D | Box | Standard / power | Raw / structured | IDs, measures, diagnostics, normalization |
| 2D | Rectangular periodic | Standard / power | Raw / structured | Edge shifts, reciprocity, wrapping |
| 3D | Box | Standard / power | Raw / structured | IDs, volumes, faces, empty cells |
| 3D | Orthorhombic periodic | Standard / power | Raw / structured | Partial periodicity and face shifts |
| 3D | Triclinic periodic | Standard / power | Raw / structured | Image labels, remapping, robustness |

Weight-first tests must include:

- equivalence to radii after a known global shift;
- invariance under adding one common constant;
- conflicting weights/radii input;
- finite-value validation;
- empty or hidden cells under extreme weights.

### Separator inverse matrix

Required representative coverage:

- exact connected recovery modulo global gauge;
- disconnected observation components and explicit offset policy;
- zero-confidence rows and effective-graph semantics;
- noisy squared and Huber fits;
- infeasible equality and interval restrictions with contradiction witnesses;
- repeated/parallel observations;
- requested periodic image versus realized other image;
- algebraic fit without realized face support;
- active-set termination, cycling, and path diagnostics;
- preferred versus compatibility imports and result adapters.

### Equivalence and performance

- preferred and compatibility APIs must agree numerically;
- structured and raw routes must describe the same cells and IDs;
- dense and optional sparse quadratic paths, if included, must agree within
  justified tolerance;
- performance work must include representative graph sizes and memory impact,
  not only microbenchmarks.

### Downstream and packaging

- chemvoro-shaped integration test;
- clean environment installation from wheel and source distribution;
- generated README/notebook synchronization;
- `mkdocs build --strict`;
- full `python tools/release_check.py`;
- supported Python-version CI and wheel builds.

## Risks and mitigations

| Risk | Consequence | Mitigation |
|---|---|---|
| A generic result abstraction is frozen too early. | Later cell-measure work requires awkward breaking changes. | Prefer an aligned access contract and concrete 2D/3D containers; keep new surfaces provisional through downstream validation. |
| Result wrapping changes performance or computes unrequested geometry. | Existing forward workflows become slower or memory-heavy. | Preserve raw routes, make optional data lazy/absent, and benchmark representative calls. |
| Namespace cleanup breaks manuscript or downstream code. | Reproducibility and adoption are harmed. | Add preferred paths through aliases/delegation; maintain compatibility tests and migration docs. |
| Gauge and disconnected-offset terminology drifts between APIs. | Users interpret non-identified quantities as physical. | Centralize metadata and tests around ADR 0002; audit guides and reports. |
| 2D/3D “symmetry” hides real capability differences. | Users receive misleading fields or unsupported behavior. | Keep explicit namespaces and dimension-specific types; align only well-defined common concepts. |
| Sparse support introduces a heavy mandatory dependency. | Minimal installation and packaging become harder. | Keep SciPy optional and make WP-06 conditional on demonstrated benefit. |
| The refactor becomes one large implementation change. | Review and regression isolation become difficult. | Use dependent issue-sized changes and characterize behavior before refactoring. |
| Lifecycle labels are assigned too optimistically. | v0.7 creates a stronger promise than downstream use supports. | Treat new contracts as provisional unless evidence supports stable status; require explicit inventory approval. |
| Docs and implementation diverge during development. | Users cannot tell what is current. | Update current docs with each accepted change; use target labels; run strict docs and generated-file checks. |
